# Decisions

本文档记录影响未来工作的设计决策。新增重大决策时，按模板追加到最上方或创建独立 ADR 文件。

## DEC-0003: 翻译工具采用“流程优先，多模型可插拔”

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-13 |
| 状态 | 已接受 |
| 背景 | 用户目标是自动化韩文到中文游戏翻译。调研显示游戏科幻域没有公开权威结论证明某个模型长期最优，且术语、上下文和 QA 是主要质量瓶颈 |
| 决策 | 工具架构以增量 diff、术语库、TM/RAG、候选生成、自动 QA、人审回写为主线；模型 provider 可替换，默认 dry-run，GPT/OpenAI 作为兜底入口 |
| 理由 | 保证没有 API key 时也能测试工程链路，同时避免把质量押注在单一模型上 |
| 影响 | 后续新增 DeepL、Papago、Google、Azure、Qwen-MT 或本地模型时，只扩展 provider，不改扫描和写回核心 |

## DEC-0004: Paratranz 项目 6860 作为术语主来源

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-13 |
| 状态 | 已接受 |
| 背景 | 用户补充 Paratranz 术语页；子任务确认项目 API 匿名可读，`source=ko`、`dest=zh-cn`，术语约 1963 条 |
| 决策 | 使用 `https://paratranz.cn/api/projects/6860/terms` 分页同步术语缓存，同时保留 CSV / JSON 离线导入 |
| 理由 | API 比 artifact/download 更适合自动缓存；artifact 匿名访问需要登录，不能作为默认路径 |
| 影响 | `make sync-glossary` 生成 `cache/glossary/paratranz-6860.json`；未来权限变化时使用离线导入 |

## DEC-0001: 长期知识放在 `docs/`，入口放在 `AGENTS.md`

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-09 |
| 状态 | 已接受 |
| 背景 | 需要开发一个专门 for AI 的开发环境，包含产品文档、技术文档、测试方案和 agent 工作上下文 |
| 决策 | 使用根目录 `AGENTS.md` 作为统一入口，使用 `docs/` 作为长期知识库 |
| 理由 | `AGENTS.md` 是轻量跨工具入口；`docs/` 对人类和 agent 都直观，适合承载产品、技术、测试和上下文文档 |
| 影响 | 后续新增长期事实优先写入 `docs/`；`AGENTS.md` 只维护索引和稳定规则 |

## DEC-0002: 初版不引入具体应用技术栈

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-09 |
| 状态 | 已接受 |
| 背景 | 当前需求重点是 AI 开发环境和文档上下文体系，不是具体产品应用 |
| 决策 | 初版只落文档骨架、模板和验证脚本，不预设前端、后端、数据库或 CLI 框架 |
| 理由 | 避免过早绑定技术栈；先让信息架构可观察、可调整 |
| 影响 | 后续真正开始实现工具或应用时，需要在 `docs/tech/architecture.md` 补充技术栈和运行架构 |
